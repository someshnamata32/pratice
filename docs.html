1 Registration Flow

What happens when a user hits POST /register

1. Parse request body

Extract username, password, email, role from the incoming JSON payload.

2. Validate required fields

If any field is missing, immediately return "msg": "missing fields".

3. Check for duplicate user

Query MongoDB for an existing user with the same username. If found, return "user already exist".

4. Hash the password (bcrypt)

bcrypt.hash(password, 10) — 10 salt rounds. This is where security happens.

5. Save user to database

Store username, hashed password, email, role in MongoDB. The raw password is never stored.

6. Send welcome email

Call mail(email, username) — Nodemailer sends a confirmation email asynchronously.

7. Respond to client

Return {"msg": "registration succesfull"}

The actual route code

// app.js — line 73
app.post('/register', async (req, res) => {
  try {
    const { username, password, email, role } = req.body;

    // 1) Check fields
    if (!username || !password || !email || !role)
      return res.json({ msg: "missing fields" });

    // 2) Check duplicate
    let checkuser = await users.findOne({ username });
    if (checkuser) return res.json({ msg: "user already exist" });

    // 3) Hash password (10 salt rounds)
    let hashedpassword = await bcrypt.hash(password, 10);

    // 4) Store user
    await users.create({ username, password: hashedpassword, email, role });

    // 5) Send email
    mail(email, username);

    res.json({ msg: "registration succesfull" });

  } catch (error) {
    res.json({ msg: error.message });
  }
});

2 User Model (Schema)

Mongoose schema that defines the shape of a user document in MongoDB

Located at models/usermodel.js

const mongoose = require('mongoose');

let userschema = new mongoose.Schema({
  username:  { type: String, required: true, unique: true },
  password:  { type: String, required: true },
  email:     { type: String, required: true, unique: true },
  role:      { type: String, enum: ["seller", "buyer"], required: true }
}, { timestamps: true });

let users = mongoose.model('users', userschema);
module.exports = users;
Note: unique: true on both username and email prevents duplicate accounts at the database level. enum restricts role to only "seller" or "buyer".

3 bcrypt — Salting & Why It Matters

What bcrypt does under the hood and why you must use it

What is a salt?

A salt is a random string appended to the password before hashing. Every user gets a different salt, so even if two users have the same password, their hashes will be completely different.

How bcrypt.hash(password, 10) works

The second argument (10) is the salt round cost factor. bcrypt internally:

  1. Generates a cryptographically random salt (16 bytes → 22 chars in base64)
  2. Appends the salt to your password: salt + password
  3. Runs the Blowfish cipher — 210 = 1024 iterations of key expansion
  4. Returns a single string containing: $2b$10$[salt][hash]
Why 10 rounds? High enough to slow down attackers, low enough to not annoy users (~50ms on modern hardware). Each extra round doubles the compute time.

Why you must salt — the Repercussions

Without bcrypt salting, every security boundary crossed becomes catastrophic:

Rainbow Tables

Attackers precompute hashes for billions of common passwords. Unsalted MD5("password123") is instantly reverseable. Salts make precomputation useless — the attacker would need a table for every possible salt.

Identical Password Leak

Without salt, users "alice" and "bob" with the same password produce the same hash. An attacker who cracks one knows the other's password too.

GPU Brute Force

Modern GPUs compute billions of MD5/SHA-256 hashes per second. bcrypt's adaptive cost (Blowfish rounds) makes each guess ~50,000× slower than SHA-256, buying critical time.

Database Breach

If your DB is leaked (SQL injection, exposed backup, misconfigured MongoDB), unsalted hashes are trivial to crack. With bcrypt + salt, cracking even one password becomes prohibitively expensive.

What happens at login?

bcrypt.compare(password, userdetails.password) extracts the salt from the stored hash, re-applies it to the incoming password, and compares.

// app.js — login route
let checkpassword = bcrypt.compare(password, userdetails.password);
if (!checkpassword) return res.json({ msg: "invalid credentials" });
Note: There's a bug here — bcrypt.compare returns a Promise, not a boolean. Needs await in your codebase.

4 What Happens Without bcrypt?

Every single way an attacker can exploit unsalted / unhashed passwords

Plain Text Passwords

If you store {username: "alice", password: "password123"} — the database has the actual password. One breach and every account is compromised instantly. Users reuse passwords across sites. You've now leaked their bank email, Netflix, etc.

This is the #1 security sin. Never, ever store plain text passwords.

Unsalted Hash (e.g. plain MD5/SHA)

Rainbow Table Attack

Free online rainbow tables (like CrackStation) index trillions of precomputed hashes. Query MD5("password123") → instant plaintext.

Dictionary Attack

Take 10,000 most common passwords, hash each one, compare against DB dump. Finds ~40% of passwords in seconds.

Mask Attack

Know the password pattern? E.g. "Summer2024" — hash variations like "Summer2025", "Summer2026" and match against the dump.

Collision / Matching

Two users with password "qwerty" produce identical hashes without salt. Crack one, compromise both.

Weak Hash + No Salt (What attackers do)

Step-by-step attack on a leaked database with unsalted hashes:

  1. Extract the password hash column from the DB dump
  2. Identify hash type (MD5 starts with $1$, SHA-256 is 64 hex chars, etc.)
  3. Run hashcat or John the Ripper on a GPU rig — MD5 runs at ~100B hashes/sec on a single RTX 4090
  4. "password123" is cracked in 0.000001 seconds
  5. Credential stuffing — try those email:password combos on Gmail, GitHub, banking sites
Bottom line: Without bcrypt, a database breach is a catastrophic event. With bcrypt + salt rounds, it's a minor incident.

5 Nodemailer Integration

Sending welcome emails via Gmail — the full pipeline

The mailer utility — utills/gmail.js

This file exports an async function that takes email and username, creates a Gmail transporter, and sends a confirmation message.

const nodemailer = require('nodemailer');

async function mail(email, username) {
  const transporter = await nodemailer.createTransport({
    service: "gmail",
    auth: {
      user: process.env.GMAILUSER,
      pass: process.env.GMAILPASS
    }
  });

  const message = {
    from: process.env.GMAILUSER,
    to: email,
    subject: "Account creation",
    text: `Hi, ${username} your account is created successfully`,
    html: "registration"
  };

  await transporter.sendMail(message);
  console.log("email sent");
}

module.exports = mail;

How to import & use it

In any route file, require the function and call it with the recipient's email and username:

// 1. Import (at the top of app.js)
let mail = require('./utills/gmail');

// 2. Call it after successful registration
mail(email, username);
Important: The function signature is mail(email, username). Email comes first, username second. The email is the destination address; the username is used in the message body.

Setup instructions (Gmail App Password)

For Gmail to allow Nodemailer, you must use an App Password (not your regular Gmail password):

  1. Go to Google Account → Security
  2. Enable 2-Step Verification (required for App Passwords)
  3. Search for "App passwords" in Google Account settings
  4. Select Mail as the app and Other as the device
  5. Copy the 16-character password generated
  6. Add to your .env file:
# .env
GMAILUSER='your.email@gmail.com'
GMAILPASS='xxxx xxxx xxxx xxxx'

How it's wired into the registration route

In app.js:88, after the user is saved to the database, the mail function is called fire-and-forget (no await):

// User saved successfully, now notify them
mail(email, username);

// Response sent back to client independently
res.json({ msg: "registration succesfull" });

This means the response is immediate — the email sends in the background.

6 Rate Limiting

Protecting your API from brute-force and abuse using express-rate-limit

The middleware — middlewares/ratelimit.js

const { rateLimit } = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  limit: 3,                  // Max 3 requests per window
  standardHeaders: 'draft-8',
  legacyHeaders: false,
  ipv6Subnet: 56
});

module.exports = limiter;
Current limit is 3 requests per 15 minutes — this is extremely strict. In production you'd typically set limit: 100 or higher for general routes, and limit: 5-10 specifically on /register and /login.

How it's applied globally

In app.js:16, the limiter is mounted as global middleware — every route is rate-limited:

// app.js
app.use(limiter);

This means POST /register, POST /login, GET /products — all share the same 3-request-per-15-minute limit per IP.

Why rate limiting matters for security

Brute-force login

Without rate limiting, an attacker can try thousands of passwords against /login in seconds. With rate limiting, they get 3 attempts per 15 minutes — infeasible.

Registration spam

Bots can create thousands of fake accounts. Rate limiting /register caps this to 3 accounts per 15 minutes per IP.

DoS amplification

Attackers can hammer your DB with expensive bcrypt operations. bcrypt is intentionally slow — rate limiting prevents resource exhaustion.

Credential stuffing

Automated tools try breached email:password pairs against your API. 3 tries per 15 minutes makes this economically unviable.

Done Right

  • bcrypt hashing with salt rounds (cost factor 10)
  • Raw password never stored in DB
  • Unique constraint on username & email
  • Role enum restricts to seller/buyer only
